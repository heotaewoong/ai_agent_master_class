# HOW I AI SCREENER — 나의 AI 워크플로우

---

# SECTION 1: 워크플로우 소개

## 1. 워크플로우 이름

**AI 뉴스레터 자동화 파이프라인**
*(AI News Automation Pipeline — LangGraph 멀티에이전트 버전)*

---

## 2. 한 줄 요약

주제 키워드 입력(입력) → 5개 AI 에이전트(Researcher·Analyst·Writer·Editor·Publisher)가 뉴스 수집·분석·집필·검수를 자동으로 수행(처리) → 브라우저에서 바로 열리는 HTML 뉴스레터 파일 자동 생성(출력)

---

## 3. 해결하는 문제

매번 AI 트렌드를 파악하기 위해 TechCrunch, MIT Tech Review, OpenAI 블로그 등 10여 개 사이트를 직접 돌아다니며 기사를 읽고, 중요한 내용을 골라 뉴스레터 초안을 손으로 작성하느라 한 호(號) 제작에 3~4시간이 소요되어 꾸준히 발행하기가 힘들었다.

---

## 4. 실사용 여부

현재 개발 완료 후 로컬에서 테스트 운영 중 (2025년 5월 기준). `python main.py --topic "AI 에이전트 최신 트렌드 2026"` 명령 하나로 전체 파이프라인 실행. 주 1~2회 실행하며 뉴스레터 초안 생성에 활용.

---

## 5. 배포 여부

현재 본인만 로컬에서 사용 중. 향후 스케줄러(cron)에 연결해 매주 자동 발행하거나, 이메일/노션으로 자동 배포하는 구조로 확장 예정.

---

# SECTION 2: 기술 소개

## 1. 사용 중인 AI 툴 / 프레임워크

| 단계 | 에이전트 | 사용 도구 |
|------|----------|-----------|
| 뉴스 수집 | Researcher | Tavily Search API (웹 검색) + feedparser (RSS 파싱) |
| 트렌드 분석 & 기사 선별 | Analyst | OpenAI GPT-4o-mini (LangChain ChatOpenAI) |
| 뉴스레터 초안 작성 | Writer | OpenAI GPT-4o-mini |
| 검수 & 퇴고 결정 | Editor | OpenAI GPT-4o-mini (조건부 엣지: revise / publish) |
| HTML 파일 출판 | Publisher | Jinja2 템플릿 엔진 |
| 워크플로우 오케스트레이션 | — | **LangGraph** (StateGraph) |

> **LangGraph의 핵심 역할**: Editor 에이전트가 품질 기준 미달 판단 시 Writer에게 재작성을 요청하는 **사이클(cycle)** 구조를 조건부 엣지(conditional edge)로 구현.

---

## 2. 연동된 외부 서비스

- **Tavily Search API**: 주제 기반 웹 검색 (최신 AI 뉴스 탐색)
- **RSS 피드**: TechCrunch AI, The Verge AI, MIT Technology Review, VentureBeat AI, AI타임스, OpenAI Blog, Anthropic News, Google AI Blog, Hugging Face Blog 등 14개 소스
- **OpenAI API**: GPT-4o-mini 모델 호출 (Analyst · Writer · Editor)
- **파일시스템**: `./output/` 디렉토리에 `.md` + `.html` 파일 자동 저장

---

## 3. 자동화 수준

**반자동** — CLI 명령 실행 후 파이프라인은 완전 자동

명령 한 줄(`python main.py`)을 실행하면 뉴스 수집 → 분석 → 집필 → 검수 → 출판까지 전 과정이 자동으로 진행된다. 사람이 직접 개입하는 부분은 오직 "어떤 주제로 뉴스레터를 만들지"를 결정하는 것뿐이다.

반자동으로 설계한 이유: 완전 자동화하면 원치 않는 주제로 뉴스레터가 생성될 수 있고, 주제 선택 자체가 큐레이션 의도를 담기 때문에 사람이 트리거를 직접 누르는 방식을 선택했다.

---

## 4. 비용

| 항목 | 비용 |
|------|------|
| OpenAI API (GPT-4o-mini) | 실행 1회당 약 원화 10~30원 수준 (입출력 토큰 합산) |
| Tavily Search API | 월 1,000 검색 쿼리까지 무료 (Free tier 사용 중) |
| RSS 피드 | 무료 |
| **합계 (월 8회 실행 기준)** | **약 100~250원/월** |

# SECTION 3: 도입 전과 후

## 1. AI 없이는?

TechCrunch, MIT Tech Review, VentureBeat 등 주요 사이트를 하나씩 열어 기사를 읽고, 중요 내용을 직접 메모하거나 복사해 노션에 붙여넣은 뒤, 그 내용을 다시 정리해 뉴스레터 형태로 글을 썼다. 사이트 탐색 1시간 + 정리 1시간 + 집필 1~2시간으로 한 편에 3~4시간이 필요했다.

---

## 2. 도입 계기

AI 트렌드를 꾸준히 학습하고 싶었지만, 정보를 모아 뉴스레터로 정리하는 반복적이고 시간 소모적인 작업이 발목을 잡았다. LangGraph를 학습하는 시점에 "이 구조를 실제 문제에 적용해보자"는 생각으로 시작했다. 특히 Editor → Writer 재작성 루프를 표현하기 위해 LangGraph의 조건부 엣지가 딱 맞았다.

---

## 3. 절약된 것

- **시간**: 뉴스레터 1편 제작 시간 3~4시간 → 약 3~5분 (명령 실행 후 대기 시간)
- **에너지**: 정보 수집과 정리의 반복적 인지 부담 거의 제로
- **규칙성**: "귀찮아서 안 하게 되는" 문제가 사라짐. 실행 한 번으로 완성

---

## 4. 초기 시행착오

**문제 1 — Editor의 퇴고 루프 무한 반복**
Editor 에이전트가 너무 까다로운 기준으로 계속 "수정 필요" 판정을 내려 파이프라인이 종료되지 않았다. `revision_count`를 State에 추가하고, 최대 수정 횟수(2회) 초과 시 강제로 publish 처리하는 로직을 추가해 해결했다.

**문제 2 — RSS 파싱 실패 시 전체 파이프라인 중단**
일부 RSS 피드가 간헐적으로 응답하지 않을 경우 예외가 전파되어 전체가 멈췄다. `try/except`로 소스별로 독립 처리하고 실패 소스는 `errors` 리스트에 기록 후 계속 진행하도록 수정했다.

**문제 3 — JSON 응답 파싱 실패**
Analyst/Writer 노드에서 LLM이 JSON 대신 마크다운 코드블록(` ```json `)으로 감싸서 응답하는 경우가 많았다. 코드블록 제거 로직을 전처리에 추가해 해결했다.

# SECTION 4: 성과 & 한계

## 1. 가장 잘 작동하는 부분

**Analyst 에이전트의 기사 선별 능력**이 기대 이상이다. 20건 이상의 기사 중에서 실제로 중요한 5~7개를 정확히 추려내고, 각 기사가 중요한 이유를 2~3줄로 요약해준다. 사람이 직접 고를 때보다 편향이 적고 다양한 각도(기술·비즈니스·규제)를 균형 있게 포함시킨다.

또한 **전체 실행 시간이 3~5분** 안에 완료되며 깔끔한 HTML 결과물이 바로 나오는 경험이 매우 만족스럽다.

---

## 2. 아직 해결 안 된 부분

- **RSS 소스 다양성**: 한국어 AI 뉴스 소스가 부족해 영어 기사 비중이 높다. 한국어 독자를 위한 뉴스레터 품질을 높이려면 한국어 소스를 더 추가해야 한다.
- **실시간성 한계**: RSS 피드 업데이트 주기에 의존하므로 속보성 뉴스 반영이 늦을 수 있다.
- **Tavily 검색 결과 품질**: 주제에 따라 관련도가 낮은 기사가 섞이는 경우가 있다. 검색 쿼리 최적화가 필요하다.

---

## 3. 실패 경험

처음 Editor 노드를 설계할 때 "엄격한 기준"으로 프롬프트를 작성했더니 항상 "수정 필요" 판정만 내리는 문제가 생겼다. Editor가 Writer를 무한히 재작성시키는 루프에 빠진 것이다. 최대 수정 횟수 제한 로직을 추가하기 전까지 API 비용이 예상보다 10배 이상 나왔다. **"AI 에이전트 루프 설계에는 반드시 탈출 조건이 필요하다"**는 교훈을 직접 경험했다.

---

## 4. 전체 만족도

**9/10** — 추천 의향 있음.

뉴스 수집·정리라는 반복 작업을 완전히 자동화했고, LangGraph의 핵심 개념(State, Node, Edge, 조건부 분기, 사이클)을 실제 문제에 적용해 학습 효과도 높았다. 1점을 뺀 이유는 아직 스케줄러 연동 및 이메일 발송이 구현되지 않아 완전 자동화에 이르지 못했기 때문이다.

# SECTION 5: 다음 단계

## 1. 다음 버전에서 추가하고 싶은 것

- **스케줄러 연동**: macOS cron 또는 GitHub Actions로 매주 월요일 자동 실행
- **이메일/노션 자동 발송**: Publisher 노드에서 완성된 뉴스레터를 이메일 또는 노션 페이지로 자동 배포
- **Human-in-the-loop**: LangGraph의 `interrupt_before` 기능으로 Editor 검수 단계에서 사람이 최종 승인 후 발행하는 흐름 추가
- **다국어 소스 확장**: 일본어, 중국어 AI 뉴스 소스 추가 및 자동 번역

---

## 2. 장기적인 방향

단순 뉴스 수집을 넘어, 개인의 관심 분야와 학습 히스토리를 기억하고 **"오늘 나에게 꼭 필요한 AI 뉴스"만 큐레이션해주는 개인화 에이전트**로 발전시키고 싶다. 장기적으로는 뉴스 소비 → 인사이트 정리 → 개인 지식베이스 축적까지 이어지는 **학습 자동화 파이프라인**이 목표다.

---

## 3. 이 과정에서 배운 것

**프레임워크보다 설계가 먼저다.**

처음엔 LangGraph 문법을 익히는 데 집중했지만, 실제로 워크플로우를 만들면서 "에이전트를 몇 개로 분리할지", "어느 지점에서 루프를 끊을지", "상태(State)에 무엇을 담을지"처럼 **설계 결정**이 코드보다 훨씬 중요하다는 것을 깨달았다.

또한 에이전트는 완벽하게 동작하지 않는다. **실패를 전제로 하는 견고한 에러 핸들링**이 프로덕션 수준 워크플로우의 핵심이라는 점도 직접 경험하면서 체감했다.
